# 流程合规与 PMO 机制

## 目标

把流程规则从“建议”提升为“可审计约束”，使 Team Lead 或 PMO 可以主动发现违规、执行漂移和协同问题，并把这些问题沉淀为机制改进。

## 角色定位

- PMO 是独立治理角色，不替代 PM、Architect、QA、Platform/SRE 的正式签字职责。
- PMO 的主要作用是“发现问题、记录问题、推动改进”，而不是代替业务或技术评审做最终决策。
- 常规情况下，PMO 不直接充当正式放行 Gate，而是输出审计结论、纠正动作和流程改进项。

## 合规检查对象

- Gate 是否被跳过
- 必签角色是否缺失
- 文档状态是否与实际阶段一致
- Todo 是否缺 owner / due / evidence
- Issue 评论是否缺关键 Gate 记录
- 异常是否已升级和关闭

## 触发机制

- 每次 PR 都应视为一次独立的合规检查触发点。
- 文档 PR 重点检查：PRD 完整性、Tech 与 PRD 一致性、QA Case Design 是否存在、评审记录是否完整。
- 代码 PR 重点检查：代码与文档一致性、测试报告是否存在、验收记录是否完整。
- 除 PR 触发外，日会、周复盘、月复盘也应继续承担汇总和复盘作用。
- 当同类问题重复出现时，PMO 必须把问题回写为 prompt、workflow、template 或角色定义的改进输入。

## 检查频率

- 每日日会：检查 blocker、延期评审、状态漂移，并汇总当天审计问题
- 每周复盘：检查 Gate 绕过、重审缺失、长期阻塞
- 每月复盘：检查流程规则是否失效、是否需要修订

## 检查结果分类

- `pass`：符合规则
- `warning`：存在偏差，但未形成下游风险
- `fail`：已形成流程违规，必须纠正

## fail 级别处理

- 补齐缺失记录
- 回退到应有 Gate
- 重新评审
- 记录违规原因与预防动作

## 分级升级规则

- `P0 / P1` 流程问题应立即通知 Team Lead。
- 一般问题可按日汇总，在统一讨论中处理。
- 合规角色不得自行关闭问题，必须把问题转成正式跟踪对象并等待责任人关闭。

## 审计输出

- 审计日期
- 检查范围
- 发现的问题
- 影响的 Issue / 文档 / Gate
- 纠正动作
- owner 与截止时间

## 建议留痕位置

- 每日问题汇总：`docs/pmo/issues/YYYY-MM-DD.md`
- 单次审计记录：`docs/memo/` 或对应主 Issue comment
- 周 / 月级复盘：对应 review 纪要

## 例外审批

只有在以下条件同时满足时，才允许临时例外：

- 明确说明为什么无法遵守标准流程
- 指定例外范围与失效时间
- 指定批准人
- 指定补偿控制措施

没有例外审批记录时，不承认任何“特殊情况”。
