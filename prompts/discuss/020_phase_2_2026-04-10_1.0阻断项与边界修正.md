4. **`004` 的 1.0 必须项是 `Issue / PR / Comment / 失败阻断` 闭环**

另外，本阶段将 `优先复用现成能力，避免重复造轮子` 正式写入 [core-principles.md](/home/work/code/AgentDevFlow/docs/governance/core-principles.md)。

## 一、独立性口径修正

### 现状问题

当前根 `README.md`、[plugins/agentdevflow/README.md](/home/work/code/AgentDevFlow/plugins/agentdevflow/README.md) 等文件，仍然大量使用：

- “从 `hedge-ai` 拆出来”
- “对照 `hedge-ai` 才能理解”

作为对外主叙述。

这会导致两个问题：

1. 违反 `003_本项目独立性的说明.md`
2. 使 1.0 自举入口依赖“先知道 hedge-ai 是什么”

### 决策

从本阶段开始，文档分成两类：

#### A. 面向使用者的入口文档

包括但不限于：

- 根 `README.md`
- `plugins/agentdevflow/README.md`
- `docs/platforms/*.md`
- `docs/README.md`

规则：

- 不再以 `hedge-ai` 作为理解前提
- 只说明 AgentDevFlow 自己解决什么问题、如何运行、如何接入
- 可以提“迁移边界”“设计来源受约束”，但不能让用户必须先理解源仓库

#### B. 面向审计的迁移文档

包括但不限于：

- `docs/migration/*`
- `docs/governance/migration-boundary-from-hedge-ai.md`
- `prompts/discuss/016*`
- `prompts/discuss/020*`

规则：

- 可以继续显式引用 `hedge-ai`
- 负责说明来源、边界、映射、审计结论

### 对后续阶段的影响

- `Phase 3` 设计平台入口时，必须使用 AgentDevFlow 自洽叙述
- `Phase 7` 收口时，需要系统修正对外文档口径

## 二、角色体系修正

### 1. `Tech Lead -> 架构师`

### 现状问题

当前仓库中存在大量：

- `Tech Lead`
- `tech-lead`
- `技术负责人`

这与用户在 `003` 中的要求不一致。

### 决策

1. **1.0 正式角色名** 使用 `架构师`
2. 在真正改文件前，允许短期保留旧文件路径作为技术债
3. 但从 `Phase 3` 开始，平台方案、角色装载、README 和计划文档都应以 `架构师` 为准

### 统一映射

- 对外角色名：`架构师`
- 执行语义：负责技术可行性、架构、Tech Spec、技术 Gate
- 旧名 `Tech Lead / 技术负责人 / tech-lead` 视为待迁移兼容名称

### 2. `Researcher` 改为可选角色

### 现状问题

当前仓库多处把 `Researcher` 当作默认角色列在团队启动顺序和平台接入建议中。

这与用户在 `003` 中的要求不一致。

### 决策

1. `Researcher` 保留，但定位改为 **研究支持 / 可选角色**
2. 只有在存在明确调研型 issue、前期方案不确定性高、需要额外证据支持时才启用
3. 默认团队不要求必启 `Researcher`

### 对后续阶段的影响

- `Phase 3` 的 Claude / Codex 启动方案里，默认团队不再强制包含 `Researcher`
- `Researcher` 相关路由要改成“按需启用”

## 四、`004` 的 1.0 边界判定

### 现状问题

`004_集成github issue及消息机制的需求.md` 提到：

- GitHub Issue / PR 全套能力
- Claude 消息通信机制
- Telegram 通信能力

如果不先做边界判定，后面很容易把 1.0 范围无限拉大。

### 判定方法

本轮按“是否直接阻断 1.0 自举”判断，而不是按“是不是有价值”判断。

### 1. 1.0 必须项

以下能力属于 **1.0 必须项**：

1. `Issue First`
   - 正式流程以 Issue 为主索引
2. `PR / 双阶段 PR`
   - 文档阶段与代码阶段分离
3. `Issue Comment Gate`
   - PRD / Tech / QA / Release 的关键结论要能回链
4. `Comment 失败必须阻断`
   - 不能把未真正落地的评论视为成功
5. `人审和关闭动作可追溯`
   - 谁确认、何时确认、是否允许进入下游，必须可追溯

### 2. 1.0 推荐项

以下能力属于 **1.0 推荐但不强制一次到位**：

1. `gh` 驱动的自动 comment / PR / issue 回写
2. 平台检查脚本或 CI 规则
3. 面向 Claude / Codex 的评论投递辅助入口

这些能力能显著提升执行度，但不是“必须全自动”才能算 1.0 自举。

### 3. 可后置项

以下能力 **不作为 1.0 阻断项**：

1. Telegram 自动通知
2. 完整消息中台或跨平台通知编排
3. 非 GitHub 宿主下的所有自动适配

理由：

- 它们会提升可用性，但不决定 Claude / Codex 能否在当前仓库中真实跑通一轮自举
- 1.0 核心目标是“真实自举”，不是“全自动通知体系一次到位”

## 五、被本阶段确认的主要阻断项

以下事项若不先修正，`Phase 3` 无法稳定定稿：

1. 入口文档的独立性口径仍未调整
2. 角色命名与默认角色体系仍未调整
3. 平台方案若不明确复用边界，容易重复发明 Team / 多 agent 机制
4. 若不先排除 Telegram 等后置能力，平台方案会被错误拉大

## 六、对 Phase 3 的输入要求

`Phase 3` 在锁定平台对等方案时，必须以本文件为前提：

1. Claude 方案必须是“复用 Agent Team + 强约束 Team Lead”
2. Codex 方案必须是“复用多 agent 调度 + 构造 Team Lead 等效编排者”
3. 默认团队不强制 `Researcher`
4. 角色口径使用 `架构师`
5. 先保障 `Issue / PR / Comment / 阻断` 闭环，再谈通知增强

## 七、Phase 2 最终判断

`Phase 2` 的目标不是把所有文件都改完，而是先锁定 1.0 自举的边界和阻断项，避免后续平台方案建立在错误前提上。

截至当前阶段，结论已经足够明确：

- 1.0 的真实主线是：`独立入口 + 正确角色体系 + 复用宿主能力 + Issue/PR/Comment 闭环`
- Telegram 与完整消息自动化不属于当前阻断项
- 可以进入 `Phase 3`，开始设计 `Claude / Codex` 的对等自举方案
