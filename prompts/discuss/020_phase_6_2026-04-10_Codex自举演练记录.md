# 020 Phase 6 - Codex 自举演练记录

## 目的

本文件记录 `Phase 6` 的 Codex 侧真实自举情况，重点回答：

1. 是否真的在 Codex 中使用了多 agent 协作，而不是主会话单独完成
2. 这轮协作做了什么真实任务
3. 得到了哪些角色视角结论
4. 最终对仓库做了哪些改动

## 一、演练任务

本轮 Codex 侧选择的真实任务是：

**继续收口 `Codex` 适配入口，使 `adapters/codex/README.md` 从旧的“接入说明”升级为真正的自举入口。**

选择这个任务的原因：

1. 它仍属于 AgentDevFlow 自己的仓库
2. 它是 `Phase 5` 之后仍然存在的真实缺口
3. 它适合用多角色协作方式验证：
   - PM 看入口是否足够让用户理解
   - 架构师看启动顺序和主会话职责是否闭环
   - QA 看验收项和关键风险是否明确

## 二、Codex 侧实际编排方式

主会话承担 `Team Lead 等效编排者` 职责，先完成：

1. 选定本轮主线任务
2. 明确目标文件：
   - `adapters/codex/README.md`
   - `adapters/codex/platform-check-example.md`
3. 派发 3 个只读子任务：
   - Product Manager 视角
   - 架构师视角
   - QA 视角

## 三、子代理返回的关键结论

### 1. QA 视角结论

QA 子代理指出：

1. 主会话必须先读取共享入口并建立 `project_id`、主 issue、当前 Gate、Todo / 状态板
2. 每个 subagent 都必须带明确读物、职责边界和写入范围启动
3. 文档阶段必须先于实现阶段，且 Issue / Comment / Human Review / Gate 都要可追溯

同时明确指出最大风险：

- `adapters/codex/README.md` 仍停留在旧版“接入说明”口径，和 `docs/platforms/codex.md`、`plugins/agentdevflow/README.md` 不一致

### 2. 架构师视角结论

架构师子代理指出：

1. `adapters/codex/README.md` 还缺明确启动顺序
2. 主会话的 `Team Lead 等效编排者` 职责不完整
3. subagent 约束没有写透，尤其缺：
   - 当前 Gate
   - 写入范围
   - 初始化确认格式

### 3. Product Manager 视角结论

PM 子代理指出：

1. 入口文档最缺的是“主会话承担 Team Lead 等效编排者”的职责说明
2. 还缺“先读什么、何时建主 issue / Gate / Todo / 状态板、何时再创建角色”的顺序
3. 还缺“角色 subagent 最小约束”
