## 结论

`skills/` 当前已经有不少内容与 `002` 对齐，但仍存在一批明确需要收口的旧流程残留。问题主要集中在三类：

1. 旧 Gate 编号残留
2. 把 `HR#1 / HR#2` 混写成 Agent 签字节点
3. 把 `文档 PR 合并后进入 Implementation` 写回“等待 Human 通知 Engineer”

## Findings

### 1. 高：`skills/agent-bootstrap/SKILL.md` 仍使用旧 Gate 定义

[skills/agent-bootstrap/SKILL.md](/home/work/code/AgentDevFlow/skills/agent-bootstrap/SKILL.md:94) 到 [skills/agent-bootstrap/SKILL.md](/home/work/code/AgentDevFlow/skills/agent-bootstrap/SKILL.md:96) 仍写：

- `Gate 3: 文档 PR 合并（Human Review #1）`
- `Gate 4: 代码 PR 合并（Human Review #2）`
- `Gate 5: Issue Comment Gate + 发布`

这与 `002` 当前定义直接冲突。这个文件必须优先修，不然会把整套 `skills` 资产重新拉回旧流程。

### 2. 高：`skills/engineer/SKILL.md` 仍把 Implementation 启动绑定到额外 Human 通知

[skills/engineer/SKILL.md](/home/work/code/AgentDevFlow/skills/engineer/SKILL.md:58) 到 [skills/engineer/SKILL.md](/home/work/code/AgentDevFlow/skills/engineer/SKILL.md:61) 仍写：

- `等待文档 PR 合并通知`
- `收到 Human 通知后开始实现（Gate 3）`

这与 `002` 当前口径冲突。现在的规则是：

- `文档 PR 合并 = 设计确认`
- Engineer 可进入 `Gate 3`
- 不再把 `Human 通知 Engineer` 写成必要前置

### 3. 高：`skills/architect/SKILL.md` 把 HR#1/HR#2 写成 Architect 签字节点

[skills/architect/SKILL.md](/home/work/code/AgentDevFlow/skills/architect/SKILL.md:54) 和 [skills/architect/SKILL.md](/home/work/code/AgentDevFlow/skills/architect/SKILL.md:56) 仍写：

- `签字（HR#1）`
- `签字（HR#2）`

这会把 Human review 与 Agent 内部评审混成一个层级。按 `002` 当前口径：

- Architect 可以参与 Gate 评审、QA Report 评审
- `HR#1 / HR#2` 本身是 Human review / merge decision
- 不能把 Architect 写成 HR 节点签字人

### 4. 高：`skills/workflows/qa-validation.md` 的 HR#2 前签字矩阵仍错

[skills/workflows/qa-validation.md](/home/work/code/AgentDevFlow/skills/workflows/qa-validation.md:64) 仍写：

- `三方（PM + Architect + QA）签字完成，方可进入 Human Review #2`

但 `002` 当前要求是：

- QA 先出测试报告
- `PM + Architect + Engineer` 三方评审测试报告并签字
- 然后进入代码 PR / `HR#2`

这是执行矩阵冲突，不是措辞问题。

### 5. 高：`skills/workflows/human-review.md` 仍把开发路径写成唯一强制顺序

[skills/workflows/human-review.md](/home/work/code/AgentDevFlow/skills/workflows/human-review.md:89) 仍写：

- `Gate 1 → Gate 2 → QA Case Design → 文档 PR（HR#1）→ Gate 3 → Gate 4 → 代码 PR（HR#2）→ Gate 5`

这会遗漏 `002` 已明确补充的“纯文档交付路径”，从而弱化：

- 纯文档交付与开发交付同等重要
- 纯文档交付也必须经过 Issue 主线与 HR#1
- 纯文档交付不经过 `Gate 3 / Gate 4 / HR#2`

### 6. 中：`skills/team-lead/SKILL.md` 仍保留旧通知链路

[skills/team-lead/SKILL.md](/home/work/code/AgentDevFlow/skills/team-lead/SKILL.md:65) 仍写：

- `Human 合并后通知 Engineer 开始实现`

这个表述会把 Team Lead 写成实现启动的必要中继节点。应改为：

- 文档 PR 合并后，Engineer 可进入 Implementation
- Team Lead 负责协调，不是额外审批门

同文件 [skills/team-lead/SKILL.md](/home/work/code/AgentDevFlow/skills/team-lead/SKILL.md:69) 的 `Human 发布后关闭 Issue` 也建议收紧，避免把 Gate 5 和 Human 操作混成一句不清晰的动作串。

### 7. 中：`skills/product-manager/SKILL.md` 仍保留“通知 Team Lead 执行文档 PR 合并”

[skills/product-manager/SKILL.md](/home/work/code/AgentDevFlow/skills/product-manager/SKILL.md:57) 和 [skills/product-manager/SKILL.md](/home/work/code/AgentDevFlow/skills/product-manager/SKILL.md:75) 仍写：

- `通知 Team Lead 执行文档 PR 合并`

这会引入一个 `002` 当前主流程里不存在的必要链路。更准确的表达应是：

- PM 发起并收敛 HR#1
- Human 决定并执行 PR 合并
- Team Lead 只做协调，不是 PR merge 的流程前置

### 8. 中：`skills/qa-engineer/SKILL.md` 仍有旧状态表述，需要再判定是否构成流程前置

[skills/qa-engineer/SKILL.md](/home/work/code/AgentDevFlow/skills/qa-engineer/SKILL.md:62) 到 [skills/qa-engineer/SKILL.md](/home/work/code/AgentDevFlow/skills/qa-engineer/SKILL.md:65) 中存在：

- `等待 Human 合并文档 PR`
- `等待 Human 合并代码 PR`

如果只是结果态描述，问题不大；如果被解读为 QA 的主流程触发前置，就应收紧成与 `002` 一致的状态表达。

[skills/pmo/SKILL.md](/home/work/code/AgentDevFlow/skills/pmo/SKILL.md:144) 仍出现 `Issue Comment Gate` 相关必读引用。
