## 基准

- 第一基准：`prompts/002_develop_pipeline.md`
- 第二基准：`prompts/discuss/032_2026-04-20_002流程错误复核.md`

## 目标

对 `skills/` 下正式规则文档做一次与 `002` 当前口径一致的收口，消除以下几类问题：

- 把 `Human Review #1 / #2` 误写成 Agent 签字节点
- 把 `Gate 3 / Gate 4 / Gate 5` 的定义写回旧版本
- 把 `文档 PR 合并后 Engineer 可进入 Implementation` 误写成必须等待额外 `Human 通知`
- 漏掉“纯文档交付与开发交付同等重要”的项目语义
- 混淆 Human 专属操作、Agent 内部签字、Issue Comment 留痕三类机制

当前仍把：

- `Gate 3` 写成 `文档 PR 合并（HR#1）`
- `Gate 4` 写成 `代码 PR 合并（HR#2）`
- `Gate 5` 写成 `Issue Comment Gate + 发布`

这与 `002` 当前定义直接冲突。必须优先修正为：

- `Gate 3 = Implementation`
- `Gate 4 = QA Validation`
- `Gate 5 = Release`
- `HR#1 / HR#2` 是 PR merge / Human review 节点，不是 Gate 编号

### 2. `skills/engineer/SKILL.md`

- `等待文档 PR 合并通知`

- `文档 PR 合并 = 设计确认`
- `Engineer 可进入 Gate 3`
- 不再把 `Human 通知 Engineer` 写成必要前置

同时需要清理把 `HR#2` 前 Agent 内部评审写成 `Architect + QA` 的旧说法。

### 3. `skills/architect/SKILL.md`

当前把：

- `文档 PR 合并` 写成 `签字（HR#1）`
- `代码 PR 合并` 写成 `签字（HR#2）`

需要改成：

- Architect 参与 Gate 内部评审或测试报告评审

### 4. `skills/workflows/qa-validation.md`

当前仍写：

- `三方（PM + Architect + QA）签字完成，方可进入 Human Review #2`

需改为与 `002` 一致：

- QA 产出测试报告
- `PM + Architect + Engineer` 三方评审测试报告并签字
- 然后进入代码 PR / `Human Review #2`

### 5. `skills/workflows/human-review.md`

当前文档仍把开发路径写成唯一主线：

- `Gate 1 -> Gate 2 -> QA Case Design -> 文档 PR（HR#1）-> Gate 3 -> Gate 4 -> 代码 PR（HR#2）-> Gate 5`

需要拆清：

- 开发交付路径
- 纯文档交付路径

并明确：

- `HR#1 / HR#2` 是 Human review / merge decision
- 不是 Agent 签字 Gate

## 统一收口口径

### 1. Gate 定义

- `Gate 1 = PRD Review`
- `Gate 2 = Tech Review`
- `Gate 3 = Implementation`
- `Gate 4 = QA Validation`
- `Gate 5 = Release`

### 2. Human Review 定义

- `HR#1 = 文档 PR 的 Human review / merge decision`
- `HR#2 = 代码 PR 的 Human review / merge decision`
- `HR#1 / HR#2` 不是 Agent 签字节点

### 3. 纯文档交付语义

- 在 AgentDevFlow 项目语义下，纯文档交付与开发交付同等重要
- 纯文档交付也必须严格经过 Issue 主线与 HR#1
- 纯文档交付不经过 `Gate 3 / Gate 4 / HR#2`

### 4. Human 专属操作

- PR 合并：Human 专属
- Issue 关闭：Human 专属

### 5. Agent 内部签字

- Gate 1：`PM + Architect + QA`
- Gate 2：`QA + Engineer + PM`
- QA Case Design：`PM + Architect + Engineer`
- QA Report 评审：`PM + Architect + Engineer`

## 修改后抽检关键词

- `Human 通知 Engineer`
- `通知 Team Lead 执行文档 PR 合并`
- `Gate 3: 文档 PR 合并`
- `Gate 4: 代码 PR 合并`

- `签字（HR#1）`
- `签字（HR#2）`
- `PM + Architect + QA`
- `QA + PM + Engineer`
- `代为关闭`

## 完成定义

完成后应满足：

- `skills/` 下不再出现与 `002` 当前 Gate 定义冲突的旧编号
- 不再把 `HR#1 / HR#2` 写成 Agent 签字节点
- 不再把 `Human 通知 Engineer` 写成实现前必要链路
- 已明确写出纯文档交付与开发交付同等重要
- 无新增 Gate、无新增审批层、无新造术语
