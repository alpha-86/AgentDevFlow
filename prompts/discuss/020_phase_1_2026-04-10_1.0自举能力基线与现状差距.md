## 目的

本文件用于执行 [020_2026-04-10_AgentDevFlow_1.0_自举里程碑执行计划](/home/work/code/AgentDevFlow/prompts/discuss/020_2026-04-10_AgentDevFlow_1.0_自举里程碑执行计划.md) 的 `Phase 1`，重新验真：

2. 当前仓库与 `016 / 019` 等已有结论的可信范围

## 结论摘要

当前仓库 **还不能判定为达到 1.0 自举**。

原因不是“文档不够多”，而是以下事实同时成立：

1. 当前完成度结论主要证明了“迁移文档与中文化收口”，不足以证明“Claude / Codex 可真实自举”。
2. `003`、`004` 对应的关键约束仍未真正收口，会直接改变 1.0 运行方案边界。
3. `Claude` 与 `Codex` 的平台文档还停留在“最小接入说明”，没有形成真实自举的启动入口、编排约束和验收证据。
4. 当前仓库仍存在和 1.0 目标冲突的内容，包括：
   - 用 `hedge-ai` 作为对外解释主线
   - 角色命名仍保留 `Tech Lead`
   - `Researcher` 仍被当作默认角色而非可选角色
   - `Claude` 适配层残留 `prompts/zh-cn/` 失效路径
   - `Codex` 插件只有元数据，没有真正的自举入口表达

## 一、按 001~005 重新提炼的 1.0 验收要求

### 1. `001_init.md`

要求的不是“把文档从 hedge-ai 拆出来”，而是把 `hedge-ai` 的通用产研机制沉淀为：

- 可被 `Claude`、`Codex`、`OpenCode` 使用的独立插件或技能包
- 有清晰依赖说明
- 有严格的 `README.md` 与 `CHANGELOG.md`
- 有适合 Agent 阅读与持续迭代的规范化结构

对 1.0 的含义：

- 必须是“可被平台真实装载和执行”的资产
- 不能只是“已经拆好的一堆文档”

### 2. `002_双PR&issue&核心原则沉淀.md`

补充了两条 1.0 主骨架要求：

- 必须把双阶段 PR、Issue 驱动、人机协同沉淀为核心原则
- 必须考虑一个项目会并行多个工作项，Issue 机制不能缺席

对 1.0 的含义：

- 自举演练不能跳过 Issue / PR / Human Review / Gate
- 需要明确“人审如何触发、何时阻断、如何留痕”

### 3. `003_本项目独立性的说明.md`

这是当前仓库和 1.0 之间最直接的冲突源之一。用户要求：

- 项目面向使用者时，不应再把 `hedge-ai` 当作理解前提
- 文档中不能用 `hedge-ai` 作为逻辑解释依据
- `Researcher` 不是必选角色
- `Tech Lead` 应改名为 `架构师`

对 1.0 的含义：

- 面向使用者的 README / 插件入口 / 平台接入文档必须独立自洽
- 角色体系会直接影响 Claude / Codex 的启动顺序和编排方案

### 4. `004_集成github issue及消息机制的需求.md`

用户要求迁移的不只是“Issue 概念”，而是：

- GitHub Issue / PR 的整套工作机制
- 与 Claude 的消息通信机制
- Telegram 通知能力
- README 中要说明这些机制解决什么问题、为什么采用这种方式

对 1.0 的含义：

- 需要明确区分：哪些是 1.0 必须真实跑通的，哪些可留作后续增强
- 但如果 1.0 完全没有可执行的 Issue / Comment / 通知承载，就不能称为自举完成

### 5. `005_AgentDevFlow_1.0_自举_里程碑目标.md`

用户已经明确了最严格口径：

- 不能假设 `001`、`002` 已正确完成
- `003`、`004` 明确还没实现
- `Claude Code` 与 `Codex` 都必须真实跑通一轮 “AgentDevFlow 迭代 AgentDevFlow”
- `Codex` 允许依赖平台已有多 agent 能力
- AgentDevFlow 必须新增重要原则：`优先复用现成能力，避免重复造轮子`

## 二、从 hedge-ai 抽取的 1.0 最小闭环能力基线

本轮只抽取与通用研发自举直接相关、且允许迁移的机制。

### A. 启动与装载

- 平台入口知道先读什么、按什么顺序创建角色
- 团队启动前必须先建立主 issue、交付目录、todo、状态板
- 角色创建不能用空白 agent，必须读取角色定义、playbook、当前 workflow

### B. 主线流程

- Issue First
- PM 先回到问题本身
- PRD 不写技术实现
- Tech Review
- QA 参与 Gate
- 双阶段 PR
- Human Review #1 / #2
- Release / 验收 / Human 关闭

### C. 留痕与阻断

- 关键结论必须回写到 Issue / Memo / Todo / Record
- Issue Comment 不是可选项
- 缺失 Gate、缺失签字、缺失产物链接时不能宣称完成

### D. 治理与恢复

- Team Lead 只做协调，不越权做签字
- PMO / Process Auditor 负责主动检查，但不替代正式评审角色
- 上下文恢复必须从 issue / gate / todo / risk 开始

### E. 平台复用原则

- 优先复用宿主平台已有能力
- 只在宿主能力不足以承载 AgentDevFlow 主干机制时，才新增本仓库自己的入口或约束

## 三、当前仓库的真实现状

### 1. 已有但只能算“规则资产”的部分

以下内容能够支持“规则层存在”，但还不能证明“平台可自举”：

- `prompts/` 已覆盖 Issue / Gate / 双阶段 PR / 人审 / 合规 / 留痕主线
- `skills/shared/` 已形成角色、workflow、template 的共享资产层
- `docs/migration/` 已形成迁移审计与边界文档
- `docs/reference/dependencies.md` 已开始讨论依赖边界

### 2. 还不能支撑 1.0 自举的部分

#### 2.1 平台入口过弱

`docs/platforms/claude-code.md` 与 `docs/platforms/codex.md` 当前只是：

- 最小安装
- 阅读建议
- 模糊角色建议

缺失：

- 自举启动顺序
- 角色装载强规则
- Team Lead 如何在平台里承担启动职责
- 自举演练任务如何执行
- 通过 / 失败判定
- 演练证据格式

#### 2.2 Claude 适配层未收口

`adapters/claude/.claude/skills/TEAM_SETUP.md` 目前仍有明显问题：

- 仍为英文
- 仍引用已删除的 `prompts/zh-cn/`
- 只有 4 条极简步骤，不足以承载真实 Team 启动

这说明 Claude 侧当前没有真正可用的 1.0 启动入口。

#### 2.3 Codex 适配层未收口

`plugins/agentdevflow/.codex-plugin/plugin.json` 当前只有基础元数据。

这只能证明“插件可被识别”，不能证明：

- 如何在 Codex 中以 AgentDevFlow 启动多 agent
- 如何映射 Team Lead 等效编排者
- 如何约束 subagent 读取共享规则
- 如何触发自举演练与人审 Gate

#### 2.4 对外叙述不满足独立性要求

根 `README.md`、插件入口 `plugins/agentdevflow/README.md` 目前都把：

- “从 hedge-ai 拆出来”
- “对照 hedge-ai”

当作主要解释逻辑。

这违背了 `003` 的独立性要求，也不适合作为 1.0 使用者入口。

#### 2.5 角色体系与用户要求不一致

当前仓库中仍然普遍存在：

- `Tech Lead`
- `Researcher` 作为默认角色

这与 `003` 的要求不一致，会直接影响平台加载顺序、默认团队组成和启动模板。

#### 2.6 004 对应机制仍未真正判定

当前仓库虽然沉淀了：

- Issue 驱动规则
- Issue Comment Gate
- PMO 主动检查

但还没有对 `004` 中要求的下列内容做出 1.0 级别的明确判定：

- GitHub Issue/PR 自动化在哪一层承载
- Claude 消息通信机制哪些是必须项
- Telegram 通知是否纳入 1.0 必需闭环
- 哪些内容可以后置，哪些会阻断 1.0 自举

#### 2.7 当前“完成度审计”口径偏乐观

`docs/migration/completion-audit.md` 与 `docs/migration/review-vs-hedge-ai.md` 的现有口径，更多证明：

- 主干机制已迁移
- 文档已收口

但对 1.0 来说，这还不等于：

- 平台可真实运行
- Claude / Codex 均可对等自举

## 四、按能力项给出的现状判定

### 必须真实跑通的能力

| 能力 | 当前判定 | 说明 |
|------|---------|------|
| Claude 团队启动入口 | 未达标 | 有零散文档，无真实自举入口 |
| Codex 多 agent 启动入口 | 未达标 | 有插件元数据，无编排入口 |
| 角色加载与必读约束 | 部分达标 | 共享资产有，但平台接入未强约束 |
| Issue First 主线 | 部分达标 | 规则存在，平台承载和演练未验证 |
| 双阶段 PR + Human Review | 部分达标 | 规则存在，未形成真实平台演练 |
| Issue Comment Gate | 部分达标 | 规则存在，1.0 承载边界未锁定 |
| PMO 主动检查 | 部分达标 | 规则存在，平台触发方式未锁定 |
| 上下文恢复 | 部分达标 | 共享规则存在，平台恢复入口未明确 |
| 对自身仓库完成一轮自举 | 未达标 | 尚无 Claude/Codex 实跑证据 |

### 可直接复用的平台或外部能力

| 能力 | 当前判断 |
|------|---------|
| Claude Agent Team 能力 | 应直接复用，不重复造 Team |
| Codex 多 agent 能力 | 应直接复用，作为对等编排宿主 |
| Git / PR / Issue 宿主能力 | 应直接复用 |
| `gh` / stacked PR / CI | 推荐复用，不应重复造轮子 |
| `gstack` / `superpowers` | 可选增强，应明确复用边界 |

### 本轮不应扩大到 1.0 的能力

- 新增 `hedge-ai` 中不存在的核心流程
- 非必要浏览器 / 设计 / 部署 / 安全能力
- 与自举无关的“顺手优化”

## 五、Phase 2 必须先处理的阻断项

以下事项不解决，平台方案无法稳定定稿：

1. 独立性口径修正
   - README / 插件入口 / 平台文档不能再以 `hedge-ai` 作为对外理解前提
2. 角色体系修正
   - `Tech Lead -> 架构师`
   - `Researcher` 调整为可选角色
3. 复用原则落地
   - 明确哪些直接复用 Claude / Codex / 外部宿主能力
4. `004` 对应边界判定
   - Issue / PR / Comment / 消息 / 通知哪些属于 1.0 必须项
5. 完成度口径纠偏
   - 不再把“迁移完成”直接写成“自举完成”

## 六、对后续阶段的建议

### 对 Phase 2 的建议

先做边界修正，再锁平台方案。

原因：

- 当前平台方案边界仍受 `003 / 004` 影响
- 如果现在直接定 Claude / Codex 的完整编排方案，后续很可能返工

### 对 Phase 3 的建议

平台方案必须产出：

- Claude 的 Team Lead 约束入口
- Codex 的多 agent 编排入口
- 两个平台共同主干与差异清单
- 自举演练启动步骤

## 七、Phase 1 最终判断

截至 `2026-04-10`，AgentDevFlow 的真实状态应表述为：

- 已形成较完整的中文规则资产和共享技能资产
- 尚未完成 1.0 自举所要求的 Claude / Codex 双平台真实运行闭环
- 目前离 1.0 最近的缺口不是“再写更多规则文档”，而是：
  - 修正边界
  - 补强平台入口
  - 明确复用策略
  - 完成双平台真实演练

因此，`Phase 1` 结论是：**继续进入 Phase 2，优先处理阻断项与边界修正，不得直接宣称 1.0 已达成。**
